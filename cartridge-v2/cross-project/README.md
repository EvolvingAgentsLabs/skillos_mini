# cartridge-from-memory (background analysis — original workspace-level framing)

A cross-project prototype that introduces a **second cartridge shape** — `memory-navigation` — alongside the existing `pipeline-with-validators` shape. The design hinges on a single empirical claim verified against the actual repos:

> The dream-engine memory already emitted by skillos / llmunix-dreamos / skillos_plugin is hierarchical, cross-linked, semantically tagged, and confidence-scored. It is **already** cartridge-shaped. We don't need to change how memory is generated. We need a runtime that treats it as a navigable knowledge base.

This prototype provides that runtime as a manifest + nav agent. It does not modify any existing memory generator. (Earlier this doc said it would not touch skillos_mini regulated cartridges either — that decision was reversed; see `skillos_mini/cartridge-v2/` for the unified approach.)

---

## 1. Why this exists

The previous design discussion reached two conclusions:

1. **For regulated outputs (electricista quotes, IEC 60364 compliance), the cartridge spine must stay deterministic.** Pipeline + JSON-Schema blackboard + Python/TS validators. The validator-on-blackboard is what makes skillos_mini safe and is the moat against Anthropic Skills / Edge Gallery commodity skills. **Do not touch.**

2. **For advisory / educational / self-knowledge domains, the better shape is a guided navigation over a hierarchical markdown knowledge tree** (Karpathy "memory as Wikipedia"). The LLM reads the current doc, decides what's relevant to the user's goal, and chooses which linked next-doc to descend into. Termination = "I have enough — emit synthesis."

Then the user's insight: **most of these domains do not need to be authored. They are already being generated.** The dream engine in skillos and llmunix-dreamos consolidates traces into level_1_epics, level_2_architecture, level_3_tactical, level_4_reactive. Each strategy is a markdown file with rich frontmatter and inline cross-links. That is exactly the shape an advisory cartridge needs.

So the cartridge becomes a **manifest** that points at a memory tree, declares its conventions, and exposes it through a navigator. No new authoring. The system's own learnings become its own cartridges.

---

## 2. The two shapes (decision matrix)

| Concern | `pipeline-with-validators` | `memory-navigation` |
|---|---|---|
| Use case | Regulated trade output (electricista PDF quote) | Advisory / self-knowledge / educational walk |
| Spine | Fixed agent sequence in `cartridge.yaml`'s `flows:` | LLM-guided walk over linked markdown tree |
| Output contract | JSON-Schema-typed blackboard + artifact (PDF) | Synthesized prose answer with citations to walked docs |
| Safety mechanism | Deterministic Python/TS validators on schema keys | Confidence scores in frontmatter + `_negative_constraints.md` cross-cited |
| Authored by | Domain expert + engineer | **Already auto-generated by dream engine** (or hand-authored knowledge base) |
| When to pick | Output must pass code rules; legal/financial liability | No artifact contract; goal is to surface relevant knowledge |
| Termination | Final agent in flow emits artifact | Navigator emits synthesis when "I have enough" signal |

These shapes coexist in one runtime. A trade cartridge stays pipeline. A self-knowledge cartridge becomes memory-navigation. Future cartridges can compose: a regulated pipeline can call a memory-navigation cartridge as a knowledge lookup tool inside one of its agents (replaces today's flat `data/*.md` context-stuffing).

## 4. The manifest spec

A memory-navigation cartridge is a single YAML file. No code, no schema directory, no validators directory — those are pipeline-cartridge concerns.

```yaml
type: memory-navigation
name: <cartridge-name>
description: <one-line>

source:
  root: <absolute-or-relative-path-to-memory-tree>
  read_only: true                    # navigator MUST NOT write to source

entry:
  index: <path-relative-to-source-root>   # e.g., index.md, project_learnings.md, health_report.md
  index_strategy: derived | explicit       # if "derived", navigator synthesizes index from frontmatter

link_conventions:
  id_pattern: "strat_L[0-9]_[a-z0-9-]+"    # how docs reference each other in prose
  resolve: by_id | by_filename | by_relative_path
  bidirectional: false                     # if true, navigator can ascend cross-links

frontmatter_schema:
  required: [id, hierarchy_level, title]
  optional: [trigger_goals, confidence, preconditions, deprecated, source_traces]

navigation:
  max_hops: 6                              # bounds latency on small LLMs
  termination_signal: "synthesis_ready"    # navigator emits this token when it stops
  prefer_higher_confidence: true           # tie-breaker between candidate next-docs

constraints_doc: <optional-path>           # e.g., _negative_constraints.md
                                            # navigator reads this once at session start

trigger_goals:                             # cartridge-level entry intents (matched against user prompt)
  - "..."
```

**Why not a `cartridge.yaml` like the trade cartridges?** Because the fields are disjoint. Pipeline cartridges have `flows:`, `blackboard_schema:`, `validators:`, `agents:`. Memory-navigation cartridges have `source:`, `entry:`, `link_conventions:`, `navigation:`. The `type:` discriminator at the top tells the runtime which loader to use.

## 5. The navigator

The navigator is a single agent prompt (markdown spec, no code). It runs in three states:

1. **Index pass**: read `entry.index`, plus the cartridge's frontmatter index of all docs (built once by walking the source tree's frontmatter — not the bodies). Match user goal against `trigger_goals` of L1 docs and the index doc itself. Pick the best match.

2. **Walk pass** (loop, max `navigation.max_hops`):
    - Read the current doc body.
    - Extract: facts relevant to user goal (kept), candidate cross-links (queued).
    - Decide: is what I have enough to answer? If yes → go to synthesis. If no → pick the highest-confidence relevant cross-link from the queue, descend.

3. **Synthesis pass**: emit answer, cite walked docs by id, surface any negative constraints from `constraints_doc` that were touched.

The navigator NEVER writes to `source.root`. If it discovers a gap (a reasonable cross-link that doesn't exist), it logs the gap as an `unmet_link_request` to its own output dir — feedback for the dream engine to consolidate later.

The full navigator spec is in `nav-agent.md`.

## 7. What the dream engine should learn from a nav session (forward-looking)

Each navigator session produces a small structured trace:

```
nav_trace_<id>:
  cartridge: <name>
  goal: <user goal>
  walked: [doc_id, doc_id, ...]
  synthesis: <text>
  unmet_links: [<requested_id_or_topic>, ...]
  hops_used: <n>
  termination_reason: synthesis_ready | max_hops | user_interrupt
```

Two feedback loops emerge naturally:

- **Frequent walks** of the same path → consolidation candidate (dream engine summarizes the path into a higher-level strategy).
- **`unmet_links`** → gap signals (dream engine, on next sweep, can consolidate traces into a new strategy that fills the gap).

This is **not implemented** in the prototype — it's the natural next iteration once the navigator is live and producing traces.

## 9. What this prototype deliberately does NOT do

- Modify any memory generator (DreamEngineAgent, MemoryConsolidationAgent, etc.).
- Modify any regulated trade cartridge in skillos_mini.
- Implement runtime code (this is a markdown-OS-style prototype: specs + manifests + a worked example, executable by an LLM following the navigator prompt).
- Wire the navigator into all four hosts as a working command. Wiring is mechanical once the spec is agreed; the prototype shows the agent prompt and one working manifest.
- Change cartridge.yaml in skillos_mini.
